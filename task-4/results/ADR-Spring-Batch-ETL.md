# Architecture Decision Record: Внедрение Spring Batch для ETL операций

## Название задачи:
Реализация ETL пайплайна для обработки складских отчетов

## Дата:
21 сентября 2025

## Контекст и проблематика
TradeWare сталкивается с критическими проблемами производительности при обработке больших объемов данных в периоды пиковой нагрузки. Текущая монолитная архитектура не справляется с обработкой до 400 000 строк в сутки, что приводит к:
- Задержкам в обновлении остатков товаров
- Зависанию отчетов и интерфейсов
- Невозможности масштабирования под растущую нагрузку
- Отсутствию механизмов мониторинга и восстановления после сбоев

## Функциональные требования

| № | Действующие лица или системы | Use Case | Описание |
|---|--------------------------------|----------|----------|
| 1 | Сотрудник склада | Загрузка CSV отчета | Пользователь загружает CSV файл с остатками через веб-интерфейс |
| 2 | Система ETL | Валидация данных | Система проверяет формат и целостность загруженных данных |
| 3 | Система ETL | Обогащение данных | Система обновляет данные о товарах информацией из справочников (промо-акции, программа лояльности) |
| 4 | Система ETL | Сохранение результатов | Система записывает обработанные данные в таблицу products |
| 5 | Система мониторинга | Отслеживание выполнения | Система логирует этапы обработки и уведомляет о результатах |
| 6 | Система ETL | Обработка ошибок | При сбое система выполняет retry операции и сохраняет необработанные данные |

## Нефункциональные требования

| № | Требование |
|---|------------|
| 1 | **Производительность**: Обработка 2000 строк за 30 секунд |
| 2 | **Масштабируемость**: Поддержка 100-150 параллельных загрузок в пиковые часы |
| 3 | **Надежность**: Механизм retry при сбоях, сохранение состояния обработки |
| 4 | **Мониторинг**: Централизованный сбор логов и метрик производительности |
| 5 | **Интеграция**: Совместимость с Java 11, WildFly, PostgreSQL, GCS |
| 6 | **Миграция**: Возможность постепенного перехода к микросервисной архитектуре |

## Решение

### Архитектура с Spring Batch

Spring Batch предоставляет проверенный фреймворк для пакетной обработки данных со следующими компонентами:

#### Компоненты решения:

1. **Job Configuration**
   - Определение ETL задачи с пошаговым выполнением
   - Управление транзакциями и checkpoint механизмы
   - Возможность перезапуска с точки сбоя

2. **ItemReader (FlatFileItemReader)**
   - Чтение CSV файлов из GCS или локального хранилища
   - Поддержка больших файлов через chunk-oriented processing
   - Встроенная валидация формата

3. **ItemProcessor (ProductItemProcessor)**
   - Бизнес-логика обогащения данных
   - Запросы к справочным таблицам (loyality_data)
   - Трансформация и валидация записей

4. **ItemWriter (JdbcBatchItemWriter)**
   - Пакетная запись в PostgreSQL
   - Оптимизация через batch inserts
   - Управление транзакциями

5. **JobRepository**
   - Сохранение метаданных выполнения
   - История запусков и статистика
   - Поддержка restart/retry механизмов

### Преимущества выбранного решения:

- **Проверенная технология**: Spring Batch широко используется в enterprise решениях
- **Интеграция с экосистемой**: Нативная поддержка Spring Boot, легкая интеграция с существующим Java стеком
- **Встроенные возможности**: Retry логика, skip политики, транзакционность из коробки
- **Масштабирование**: Поддержка партиционирования и параллельной обработки
- **Мониторинг**: Интеграция с Spring Boot Actuator, метрики через Micrometer

## Альтернативы

### 1. Apache Spark
**Преимущества:**
- Высокая производительность для больших данных
- Распределенная обработка
- Поддержка streaming и batch

**Недостатки:**
- Избыточен для текущих объемов (400K записей)
- Требует отдельной инфраструктуры (Spark cluster)
- Сложность интеграции с существующим Java монолитом
- Высокий порог входа для команды

### 2. Apache Flink
**Преимущества:**
- Real-time обработка потоков
- Низкая латентность
- Exactly-once семантика

**Недостатки:**
- Ориентирован на stream processing (избыточен для batch)
- Требует отдельного кластера
- Сложность развертывания и управления

### 3. Apache Airflow
**Преимущества:**
- Оркестрация сложных пайплайнов
- Визуальный интерфейс управления
- Гибкая система планирования

**Недостатки:**
- Python-based (не интегрируется с Java кодом напрямую)
- Больше подходит для оркестрации, чем для ETL
- Требует дополнительной инфраструктуры

### 4. Apache NiFi
**Преимущества:**
- Визуальное проектирование потоков данных
- Поддержка множества источников/приемников
- Web-based интерфейс управления

**Недостатки:**
- Отдельная runtime среда
- Сложность интеграции с бизнес-логикой Java
- Избыточен для простых ETL операций

## Недостатки, ограничения, риски

### Недостатки Spring Batch:
1. **Монолитность**: Spring Batch работает внутри одного JVM процесса
2. **Ограниченное горизонтальное масштабирование**: Требует дополнительной настройки для multi-node обработки
3. **Зависимость от RDBMS**: JobRepository требует реляционную БД для метаданных

### Ограничения:
1. **Размер chunk**: Необходима тонкая настройка для оптимальной производительности
2. **Memory footprint**: При больших chunk размерах возможны проблемы с памятью
3. **Синхронная обработка**: По умолчанию обработка последовательная внутри step

### Риски:
1. **Vendor lock-in**: Привязка к Spring экосистеме
2. **Сложность отладки**: Debugging распределенных транзакций может быть сложным
3. **Производительность при экстремальных нагрузках**: Может потребоваться переход на более специализированные решения при росте до миллионов записей

### Стратегии митигации:
1. **Партиционирование**: Использование Spring Batch partitioning для параллельной обработки
2. **Оптимизация chunk size**: Динамическая настройка размера пакета
3. **Кэширование**: Использование in-memory кэша для справочных данных
4. **Асинхронная обработка**: Внедрение async processors где возможно
5. **Подготовка к миграции**: Абстрагирование бизнес-логики для возможного перехода на другие решения
