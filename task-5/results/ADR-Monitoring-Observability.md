# ADR: Мониторинг и Observability для Spring Batch ETL

## Статус
Принято

## Контекст
TradeWare внедрила Spring Batch ETL решение для обработки данных о продуктах с обогащением данными лояльности. Теперь необходимо обеспечить полную observability системы для:
- Отслеживания производительности и состояния batch jobs
- Быстрого обнаружения и диагностики проблем
- Соответствия SLA по времени обработки
- Анализа бизнес-метрик и трендов
- Обеспечения проактивного реагирования на инциденты

## Решение

### Выбранный стек технологий

#### 1. Метрики: Prometheus + Grafana
**Обоснование:**
- **Prometheus** - де-факто стандарт для сбора метрик в cloud-native приложениях
- Pull-модель идеально подходит для динамической инфраструктуры
- Мощный язык запросов PromQL для сложной аналитики
- Встроенная поддержка в Spring Boot через Micrometer
- **Grafana** - лучший инструмент визуализации с богатой экосистемой дашбордов
- Бесплатные и open source решения

#### 2. Логирование: ELK Stack (Elasticsearch + Logstash + Kibana) + Filebeat
**Обоснование:**
- **Elasticsearch** - масштабируемое хранилище для структурированных логов
- Полнотекстовый поиск и агрегации для анализа инцидентов
- **Logstash** - мощная обработка и трансформация логов
- **Kibana** - интуитивный интерфейс для анализа логов
- **Filebeat** - легковесный агент для сбора логов
- Структурированное логирование в JSON формате для лучшей индексации

#### 3. Алертинг: AlertManager
**Обоснование:**
- Нативная интеграция с Prometheus
- Группировка и дедупликация алертов
- Маршрутизация по severity levels
- Множественные каналы оповещений (email, Slack, PagerDuty)

### Альтернативы, которые рассматривались

#### 1. New Relic / Datadog / AppDynamics
**Почему не выбрали:**
- Высокая стоимость лицензий ($100+ на хост/месяц)
- Vendor lock-in
- Данные хранятся у третьей стороны
- Ограничения по retention периоду в базовых планах

#### 2. CloudWatch (AWS)
**Почему не выбрали:**
- Привязка к AWS экосистеме
- Ограниченные возможности визуализации
- Дорогое хранение метрик при большом объеме

#### 3. Splunk
**Почему не выбрали:**
- Очень высокая стоимость лицензий
- Сложность развертывания и администрирования
- Избыточен для текущих требований

#### 4. Jaeger/Zipkin для трейсинга
**Почему отложили:**
- Текущая архитектура (монолит) не требует distributed tracing
- Можно добавить позже при переходе к микросервисам
- Дополнительная сложность без явной выгоды сейчас

## Архитектурные решения

### 1. Структурированное логирование
```json
{
  "@timestamp": "2025-09-22T10:15:00.000Z",
  "level": "INFO",
  "service": "spring-batch-etl",
  "job_id": "importProductJob",
  "product_id": 12345,
  "message": "Processing product",
  "duration_ms": 25
}
```
**Преимущества:**
- Машиночитаемый формат
- Легкая фильтрация и агрегация
- Корреляция событий через trace_id

### 2. Многоуровневые метрики
- **Business метрики**: items processed, loyalty updates, processing time
- **System метрики**: JVM heap, GC, threads
- **Infrastructure метрики**: CPU, memory, disk I/O

### 3. Алертинг по уровням критичности
- **Critical**: Немедленное реагирование (PagerDuty)
- **Warning**: Уведомление в рабочее время (Slack)
- **Info**: Логирование для анализа

## Последствия

### Положительные
1. **Полная observability**: метрики + логи + алерты в едином стеке
2. **Быстрая диагностика**: структурированные данные и мощный поиск
3. **Проактивный мониторинг**: алерты до возникновения критических проблем
4. **Историческийализ**: долгосрочное хранение для выявления трендов
5. **Масштабируемость**: горизонтальное масштабирование всех компонентов
6. **Cost-effective**: open source решения без лицензионных отчислений
7. **Независимость**: контроль над данными и инфраструктурой

### Отрицательные
1. **Сложность развертывания**: 7+ компонентов для полного стека
2. **Требования к ресурсам**: ~4GB RAM минимум для всего стека
3. **Необходимость экспертизы**: требуется знание множества технологий
4. **Maintenance overhead**: обновления, бэкапы, мониторинг самого мониторинга

### Риски и митигация
1. **Риск**: Потеря метрик при сбое Prometheus
   **Митигация**: Настройка federation и резервного хранилища

2. **Риск**: Переполнение диска Elasticsearch
   **Митигация**: Retention policies и curator для автоочистки

3. **Риск**: Alert fatigue от избыточных оповещений
   **Митигация**: Тщательная настройка thresholds и группировка

## Метрики успеха
1. **MTTD (Mean Time To Detect)** < 1 минута
2. **MTTR (Mean Time To Resolution)** < 15 минут
3. **Доступность системы мониторинга** > 99.9%
4. **False positive rate** < 5%
5. **Покрытие критических метрик** = 100%

## Заключение
Выбранный стек Prometheus + Grafana + ELK + AlertManager обеспечивает полную observability для Spring Batch ETL системы с оптимальным балансом между функциональностью, стоимостью и сложностью. Решение масштабируемо и может эволюционировать вместе с архитектурой системы.
